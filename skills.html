01 Le problème commun

Pourquoi tous ces outils existent

Un harnais est livré « générique ». Il ne connaît ni votre projet, ni vos conventions, ni les habitudes de votre équipe. Si vous voulez que l'agent suive vos règles, vous avez deux choix.

Option 1 : tout retaper, à chaque fois

Au début de chaque conversation, vous rappelez à l'agent vos conventions, votre stack, ce qu'il ne doit pas toucher. Ça marche pour deux ou trois règles. Au-delà, vous oubliez, vous abrégez, et l'agent code à côté de la plaque.

Vos prompts, en boucle « Utilise du camelCase pour les variables… utilise TypeScript strict… n'écris jamais dans le dossier /legacy… nos tests sont en Vitest, pas en Jest… »

Option 2 : codifier, une fois pour toutes

Vous écrivez vos conventions dans des fichiers que le harnais charge automatiquement. L'agent les voit à chaque session, sans que vous ayez à les retaper. Vos collègues les voient aussi. C'est versionné dans Git.

Un fichier dans votre projet .claude/skills/conventions/SKILL.md
.cursor/rules/typescript.mdc
.github/copilot-instructions.md
AGENTS.md

Tous les harnais sérieux ont implémenté la deuxième option. Mais chacun à sa sauce, avec son vocabulaire propre. C'est cette diversité qu'on va démêler.

02 Le concept canonique

Les Skills d'Anthropic, en détail

Commençons par le concept tel qu'Anthropic l'a formalisé en octobre 2025 : un standard propre, simple à comprendre, et — fait remarquable — passé en deux mois au statut de standard ouvert du secteur.

Structure d'un Skill
mon-skill/
├── SKILL.md   # obligatoire
│   ├── frontmatter YAML (nom + description)
│   └── instructions Markdown
├── scripts/   # code exécutable, optionnel
├── references/  # docs détaillées, optionnel
└── assets/    # templates, fonts, etc.

Un dossier, point.

Un Skill, c'est tout simplement un dossier avec un fichier SKILL.md à la racine. Le fichier comporte deux parties : un frontmatter YAML avec un nom et une description, puis du Markdown ordinaire qui contient les instructions.

Optionnellement, le dossier peut contenir des scripts exécutables, des fichiers de référence supplémentaires, et des assets (templates, polices, etc.). Tout ça reste sur disque tant que ce n'est pas explicitement utilisé.

--- name: commit-conventions description: Règles pour rédiger des messages de commit Git.     Utiliser dès qu'il faut commiter du code, ou rédiger     un message de PR. --- # Conventions de commit Format : type(scope): description - type : feat, fix, refactor, docs, test, chore - scope : optionnel, en minuscules - description : impératif, sans point final - max 72 caractères

L'astuce : la progressive disclosure (le « dévoilement progressif »)

C'est la mécanique qui fait toute l'élégance des Skills. Trois niveaux de chargement, du plus léger au plus complet. Cliquez les phases pour voir comment ça se passe.

Phase 1 — Découverte. Au démarrage, le harnais ne charge dans le contexte du LLM que les métadonnées de chaque skill (nom + description). Léger, rapide. Le LLM sait juste qu'ils existent.

Qui fait quoi ?

Le mécanisme repose sur une division claire des rôles entre le harnais (la mécanique) et le LLM (le décideur).

Le harnais — la mécanique

Il met le système en place et exécute les opérations sur le système de fichiers. Il ne « comprend » pas le contenu — il le manipule.

  • Scanne les dossiers de skills au démarrage
  • Injecte les métadonnées (nom + description) dans le contexte initial du LLM
  • Fournit au LLM les outils bash / read nécessaires pour lire les fichiers
  • Exécute les scripts quand le LLM le demande, et lui rend la sortie

Le LLM — le décideur

Il lit les descriptions de skills disponibles et décide, à chaque tour, lesquels sont pertinents pour la requête en cours.

  • Lit les descriptions des skills disponibles dans son contexte
  • Juge si une description correspond à la tâche demandée
  • Demande au harnais de lire un SKILL.md via l'outil bash
  • Décide si des fichiers référencés (forms.md, schéma, etc.) sont nécessaires, et les lit aussi

Un standard ouvert : agentskills.io

D'abord lancée en octobre 2025 comme format propre à Anthropic, la spécification est passée en standard ouvert dès décembre 2025, confiée à la Linux Foundation. Résultat : le format SKILL.md est aujourd'hui adopté par Microsoft, OpenAI, Cursor, Goose, Amp, OpenCode, Gemini CLI… Un Skill écrit pour Claude fonctionne sur la plupart des autres harnais.

03 La grille de lecture

Trois axes pour comprendre tous les mécanismes

Les Skills d'Anthropic ne sont qu'un point dans un espace plus large. Avant d'entrer dans le panorama des outils, voici trois axes qui permettent de comparer n'importe quels deux mécanismes — chez Anthropic, Antigravity, Cursor, ou ailleurs.

01

Toujours actif / Chargé à la demande

Est-ce que le contenu est tout le temps dans le contexte du LLM, ou est-ce qu'il n'est chargé que quand pertinent ? C'est la différence fondamentale entre une « Rule » et un « Skill ».

Toujours actif

Le contenu vit dans le contexte

Chargé au démarrage de chaque session. Le LLM le voit en permanence. Pratique pour les conventions universelles. Mais ça consomme des tokens à chaque tour, qu'on en ait besoin ou pas.

CLAUDE.md, AGENTS.md, .cursorrules, Rules

⇆
À la demande

Le contenu reste sur disque

Le LLM ne voit que la description au démarrage. Il charge le contenu complet seulement si la tâche le justifie. Bien plus économe en tokens, surtout avec beaucoup de skills installés.

Skills (Anthropic, Codex, Antigravity)

02

L'utilisateur décide / L'agent décide

Qui déclenche le mécanisme ? Est-ce l'utilisateur qui invoque explicitement (/test, /deploy), ou l'agent qui juge lui-même que c'est pertinent en lisant la description ?

User-triggered

Slash command, invocation explicite

L'utilisateur tape /cmd et l'agent exécute. Pas d'ambiguïté, ça part toujours à la demande. Bon pour les workflows ponctuels qu'on ne veut pas voir se déclencher tout seul.

Workflows, Commands, Prompt files

⇆
Agent-triggered

L'agent juge lui-même la pertinence

L'agent lit les descriptions disponibles et décide tout seul, à chaque tour, si l'une d'elles correspond à la tâche. Pas besoin de penser à l'invoquer, mais ça suppose des descriptions bien rédigées.

Skills, Agent Requested rules (Cursor)

Exemple parlant

Pourquoi « l'agent décide » est plus économe

Imaginons une règle pour nommer mes fichiers en camelCase. Si elle est toujours active, elle entre dans le contexte à chaque tour — y compris quand je demande à l'agent d'éditer une classe (où la règle ne sert à rien).

Si c'est l'agent qui décide, il lit la description (« règle de nommage de fichiers ») et la charge uniquement quand on parle de fichiers. Pour ma classe, il l'ignore. Multiplié par toutes les règles du projet, l'économie de tokens est massive.

C'est exactement la philosophie d'Antigravity avec ses Skills, et celle de Cursor avec son mode « Agent Requested ».

03

Fichier monolithique / Progressive disclosure

Quand le contenu est chargé, est-ce qu'il l'est en un bloc, ou par paliers, en chargeant seulement les sous-parties pertinentes ?

Monolithique

Tout ou rien

Le fichier est chargé en entier. Simple à comprendre, mais quand le fichier devient gros, on consomme des tokens même pour les parties non utilisées dans la tâche en cours.

CLAUDE.md, .cursorrules legacy

⇆
Progressive

En 3 niveaux

D'abord les métadonnées, puis le SKILL.md, puis les fichiers référencés à la demande. Permet de packager des skills très riches sans payer le coût en tokens si on ne s'en sert pas.

Skills (toutes implémentations)

Avec ces trois axes, vous pouvez maintenant cataloguer n'importe quel mécanisme proposé par n'importe quel harnais. C'est ce qu'on va faire dans la section suivante.

04 Outil par outil

Le panorama complet

Chaque harnais a sa propre palette de mécanismes. Pour les comparer, on les classe selon les couleurs des trois familles principales : Rules (toujours actifs), Skills (à la demande, agent décide), Workflows (à la demande, user décide), Hooks (déterministes).

GitHub Copilot Microsoft
L'architecture la plus stratifiée — six familles distinctes pour des rôles précis.
Instruction
.github/copilot-instructions.md Repository-wide, toujours actif. Plus .github/instructions/*.instructions.md avec applyTo glob pour des règles ciblées.
Prompt file
.github/prompts/*.prompt.md Slash commands user-triggered. Frontmatter avec agent, tools, model.
Custom Agent
.github/agents/*.agent.md Personas spécialisées avec un set d'outils défini (« reviewer Python », « architect frontend »…).
Skill
Format SKILL.md standard Workflows réutilisables avec scripts et templates bundled. Compatible avec le standard ouvert.
Hook
Hard-stop enforcement Pour les politiques qui ne doivent jamais être contournées par le modèle.
Reconnaît nativement AGENTS.md en plus de son format propre. Les deux peuvent coexister.
Claude Code Anthropic
Le concept canonique. Anthropic a posé le standard, les autres ont suivi.

Command

La trinité la plus claire : Rules, Skills et Workflows nettement séparés.

05 Une lingua franca émerge

L'autre standard ouvert : AGENTS.md

À ce stade, vous pourriez croire que SKILL.md couvre tout. Pas tout à fait. À côté du standard des Skills, un second standard ouvert s'est imposé pour un rôle complémentaire, pas concurrent : AGENTS.md, le « README » que les agents lisent au démarrage du projet.

AGENTS.md
le contexte permanent
Un par projet, toujours chargé. Décrit la stack, les conventions, les garde-fous propres au repo.
SKILL.md
le savoir-faire à la demande
Plusieurs par projet, chargés seulement quand le LLM en a besoin. Une capacité réutilisable.

« Un README pour les agents »

Origine : OpenAI a créé le format avec Codex CLI en août 2025. Confié fin 2025 à la Linux Foundation / Agentic AI Foundation comme standard ouvert. La métaphore officielle : « think of it as a README for agents ».

Pas de syntaxe particulière, pas de schéma à apprendre :

  • Un fichier AGENTS.md au root du repo
  • Lu automatiquement par les agents au démarrage
  • Markdown libre — décrit la stack, les conventions, les contraintes
  • Hiérarchique : un par sous-dossier si besoin (le plus proche prime)
  • Versionné dans Git, partagé avec toute l'équipe

Détail qui surprend souvent : c'est l'agent lui-même qui le rédige, dans la quasi-totalité des cas. On lui demande d'explorer le repo et de produire un premier AGENTS.md, qu'on relit ensuite et qu'on commit. Le fichier reste sous responsabilité humaine, mais on part rarement d'une page blanche.

Adopté nativement par
AGENTS.md
la lingua franca des agents
Codex (OpenAI) GitHub Copilot Cursor Jules Devin Aider goose opencode Zed Warp VS Code Junie Amp RooCode Gemini CLI Cline Augment Code Factory Kilo Code UiPath Autopilot
88 Anecdote parlante : le repo OpenAI Codex contient 88 fichiers AGENTS.md à différents niveaux. Quand on dit que le format est hiérarchique, ce n'est pas anecdotique — chaque sous-projet peut avoir son propre AGENTS.md, et celui le plus proche du fichier édité prime.

Quelques outils gardent leur format propre — Claude Code avec CLAUDE.md, Gemini CLI avec GEMINI.md — sans adopter AGENTS.md. En pratique, beaucoup d'équipes mettent les règles communes dans AGENTS.md et les spécificités outil-par-outil dans les fichiers maison.

06 Pour conclure

La règle qui explique tout

Vous avez maintenant traversé un zoo de noms : Skills, Rules, Workflows, Commands, Instructions, Custom Agents, Hooks, Guidelines, Conventions… et derrière, à chaque fois, le même besoin et la même solution.

Tous ces mécanismes sont des manières de pré-charger du contexte dans le harnais, sans avoir à le retaper à chaque conversation. La compétition entre vendors se joue sur l'ergonomie — la lourdeur du fichier, la granularité du déclenchement, l'économie de tokens, la composabilité — pas sur le concept.

L'innovation pédagogiquement intéressante de ces derniers mois, c'est la progressive disclosure popularisée par les Skills d'Anthropic et reprise par Antigravity, Codex et Gemini CLI : permettre à l'agent de juger lui-même ce dont il a besoin, plutôt que d'imposer le contexte d'office. C'est cette idée qui change le plus l'expérience à grande échelle.

À retenir, en trois phrases

Au-delà des noms, tous ces mécanismes répondent à la même question : « comment customiser un harnais générique sans tout retaper ? ».

Ils se classent sur trois axes : toujours-actif vs à la demande, l'utilisateur décide vs l'agent décide, et monolithique vs progressive.

Et un standard ouvert émerge — AGENTS.md et SKILL.md — qui devrait, à terme, faire converger l'écosystème.
