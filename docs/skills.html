01 Le problème commun

Pourquoi tous ces outils existent

Un harnais — le programme qui fait tourner l'agent autour du LLM (Claude Code, Cursor, Copilot…) — est livré « générique ». Il ne connaît ni votre projet, ni vos conventions, ni les habitudes de votre équipe. Si vous voulez que l'agent suive vos règles, vous avez deux choix.

Option 1 : tout retaper, à chaque fois

Au début de chaque conversation, vous rappelez à l'agent vos conventions, votre stack, ce qu'il ne doit pas toucher. Ça marche pour deux ou trois règles. Au-delà, vous oubliez, vous abrégez, et l'agent code à côté de la plaque.

Vos prompts, en boucle « Utilise du camelCase pour les variables… utilise TypeScript strict… n'écris jamais dans le dossier /legacy… nos tests sont en Vitest, pas en Jest… »

Option 2 : codifier, une fois pour toutes

Vous écrivez vos conventions dans des fichiers que le harnais charge automatiquement. L'agent les voit à chaque session, sans que vous ayez à les retaper. Vos collègues les voient aussi. C'est versionné dans Git.

Un fichier dans votre projet .claude/skills/conventions/SKILL.md
.cursor/rules/typescript.mdc
.github/copilot-instructions.md
AGENTS.md

Tous les harnais sérieux ont implémenté la deuxième option. Mais chacun à sa sauce, avec son vocabulaire propre. C'est cette diversité qu'on va démêler.

02 Le concept canonique

Les Skills d'Anthropic, en détail

Commençons par le concept tel qu'Anthropic l'a formalisé en octobre 2025 : un standard propre, simple à comprendre, et (fait remarquable) ouvert quelques mois plus tard et repris rapidement par d'autres harnais.

Structure d'un Skill
mon-skill/
├── SKILL.md   # obligatoire
│   ├── frontmatter YAML (nom + description)
│   └── instructions Markdown
├── scripts/   # code exécutable, optionnel
├── references/  # docs détaillées, optionnel
└── assets/    # templates, fonts, etc.

Un dossier, point.

Un Skill, c'est tout simplement un dossier avec un fichier SKILL.md à la racine. Le fichier comporte deux parties : un frontmatter YAML avec un nom et une description, puis du Markdown ordinaire qui contient les instructions.

En option, le dossier peut contenir des scripts exécutables, des fichiers de référence supplémentaires, et des assets (templates, polices, etc.). Tout ça reste sur disque tant que ce n'est pas explicitement utilisé.

--- name: commit-conventions description: Règles pour rédiger des messages de commit Git.     Utiliser dès qu'il faut commiter du code, ou rédiger     un message de PR. --- # Conventions de commit Format : type(scope): description - type : feat, fix, refactor, docs, test, chore - scope : optionnel, en minuscules - description : impératif, sans point final - max 72 caractères

Et le code ? Un skill peut embarquer des scripts

Un skill n'est pas qu'un texte d'instructions. Son dossier peut aussi contenir du code prêt à l'emploi. L'intérêt : certaines tâches sont mieux faites par un programme que par le modèle — générer un classeur Excel, remplir un formulaire PDF, retailler des images en lot… Le skill livre alors un petit script (souvent en Python), et le SKILL.md indique quand s'en servir.

Côté coulisses, rien d'exotique : l'agent lance ce script comme une commande ordinaire, récupère le résultat, et poursuit. Pas de moteur spécial — du code normal, exécuté dans l'environnement du projet.

Sans script

Le modèle fabrique le fichier à la main, cellule par cellule. Lent, coûteux en tokens (les unités de texte que le LLM lit et facture), et facile à rater sur un gros document.

Avec script

Une commande, et le programme produit le fichier d'un coup, toujours de la même façon. Le modèle orchestre, le code exécute.

À retenir : un skill peut empaqueter à la fois le savoir (les instructions) et le savoir-faire outillé (les scripts) — le tout dans un simple dossier, réutilisable d'un projet à l'autre.

Qui fait quoi ?

Le mécanisme repose sur une division claire des rôles entre le harnais (la mécanique) et le LLM (le décideur).

Le harnais — la mécanique

Il met le système en place et exécute les opérations sur le système de fichiers. Il ne « comprend » pas le contenu, il le manipule.

  • Scanne les dossiers de skills au démarrage
  • Injecte les métadonnées (nom + description) dans le contexte initial du LLM
  • Fournit au LLM les outils bash / read nécessaires pour lire les fichiers
  • Quand le LLM lance un script via bash, exécute la commande shell et lui rend la sortie

Le LLM — le décideur

Il lit les descriptions de skills disponibles et décide, à chaque tour, lesquels sont pertinents pour la requête en cours.

  • Lit les descriptions des skills disponibles dans son contexte
  • Juge si une description correspond à la tâche demandée
  • Demande au harnais de lire un SKILL.md via l'outil bash
  • Décide si des fichiers référencés (forms.md, schéma, etc.) sont nécessaires, et les lit aussi

Un standard ouvert : agentskills.io

D'abord lancée en octobre 2025 comme format propre à Anthropic, la spécification a été ouverte en décembre 2025 (publiée sur agentskills.io) ; sa gouvernance long terme — possiblement au sein de l'Agentic AI Foundation, sous la Linux Foundation — reste à préciser. Plusieurs harnais ont depuis annoncé l'adoption du format (Codex, Cursor, Gemini CLI, goose, Amp…) : un Skill écrit pour Claude vise la portabilité entre harnais compatibles.

On a vu de quoi un Skill est fait (un dossier, du Markdown, des scripts). Reste la pièce maîtresse : comment tout cela entre dans le contexte sans le saturer. C'est la progressive disclosure — et elle mérite son propre chapitre.

03 Le dévoilement progressif

La progressive disclosure (le dévoilement progressif), pas à pas

C'est la mécanique qui fait toute l'élégance des Skills, et l'innovation que les autres harnais ont reprise. L'idée : ne charger dans le contexte du LLM que ce qui est réellement utile, au moment où ça l'est. Suivons concrètement ce qui se concatène au prompt, phase après phase.

Phase 1 — Découverte. Au démarrage, le harnais ne concatène au contexte que les métadonnées de chaque skill (nom + description). Le LLM sait qu'ils existent, rien de plus. Coût : ~60 tokens, quel que soit le nombre de skills installés.
Chiffres illustratifs — à rapporter à une fenêtre de contexte de l'ordre de 200 000 tokens.

Trois niveaux, un seul principe : le LLM ne paie en tokens que ce qu'il décide de lire. On peut packager des skills très riches sans alourdir chaque conversation.

04 Le contexte permanent

Le fichier de contexte du projet : AGENTS.md

Les Skills (SKILL.md) couvrent le savoir-faire à la demande qu'on vient de voir. Reste le besoin le plus courant : donner à l'agent le contexte permanent du projet — sa stack, ses conventions, ses garde-fous — dès qu'il arrive. C'est le rôle d'AGENTS.md : un simple fichier Markdown, à la racine du dépôt, que l'agent lit au démarrage. Complémentaire des Skills, pas concurrent.

# AGENTS.md ## Stack - React 18 + TypeScript strict, build via Vite - Tests : Vitest (jamais Jest) ## Conventions - Variables en camelCase, composants en PascalCase - Ne jamais écrire dans /legacy ## Commandes - Installer : pnpm install - Lancer les tests : pnpm test

« Un README pour les agents »

Aucune syntaxe à apprendre : du Markdown ordinaire, que l'agent lit en arrivant sur le projet. La métaphore officielle du standard : « think of it as a README for agents ». Rien de plus.

  • Un fichier AGENTS.md à la racine du dépôt
  • Lu automatiquement par l'agent au démarrage
  • Décrit la stack, les conventions, les commandes, les garde-fous
  • Hiérarchique : un par sous-dossier si besoin (le plus proche prime)
  • Versionné dans Git, donc partagé avec toute l'équipe
  • En général, c'est l'agent qu'on charge de le rédiger, puis on relit et on commit

Deux fichiers, deux rôles

À ne pas confondre : ils sont complémentaires, pas concurrents.

AGENTS.md

Le contexte permanent

Un par projet, toujours chargé. Ce que l'agent doit savoir en permanence sur ce dépôt : stack, conventions, interdits.

SKILL.md

Le savoir-faire à la demande

Plusieurs par projet, chargés seulement quand c'est utile. Une capacité réutilisable (générer un Excel, suivre une procédure de release…).

Vous êtes sur GitHub Copilot ? Vous l'avez déjà presque.

Le format maison de Copilot, c'est .github/copilot-instructions.md : exactement la même idée, sous un autre nom. Et Copilot reconnaît aussi AGENTS.md nativement. C'est concrètement de cela qu'il s'agit quand on parle d'agent : du contexte qu'on écrit une fois pour que l'agent arrête de tout redemander.

AGENTS.md
la « lingua franca » des agents
Reconnu nativement par
GitHub Copilot Codex (OpenAI) Cursor Gemini CLI Devin Jules Aider goose Zed VS Code RooCode Cline … et bien d'autres

Quelques outils gardent un nom propre — Claude Code avec CLAUDE.md, Gemini CLI avec GEMINI.md — mais c'est le même fichier, le même rôle. En pratique : les règles communes dans AGENTS.md, les spécificités d'un outil dans son fichier maison.

05 La grille de lecture

Trois axes pour comprendre tous les mécanismes

Les Skills et AGENTS.md ne sont que deux points dans un espace plus large. Avant d'entrer dans le panorama des outils, voici trois axes qui permettent de situer et comparer deux mécanismes quelconques, d'un harnais à l'autre.

01

Toujours actif / Chargé à la demande

Est-ce que le contenu est tout le temps dans le contexte du LLM, ou est-ce qu'il n'est chargé que quand pertinent ? C'est le premier arbitrage : confort permanent contre économie de tokens.

Toujours actif

Le contenu vit dans le contexte

Chargé au démarrage de chaque session. Le LLM le voit en permanence. Pratique pour les conventions universelles. Mais ça consomme des tokens à chaque tour, qu'on en ait besoin ou pas.

CLAUDE.md, AGENTS.md, .cursorrules, Rules

⇆
À la demande

Le contenu reste sur disque

Le LLM ne voit que la description au démarrage. Il charge le contenu complet seulement si la tâche le justifie. Bien plus économe en tokens, surtout avec beaucoup de skills installés.

Skills (Anthropic, Codex, Antigravity)

02

L'utilisateur décide / L'agent décide

Qui déclenche le mécanisme ? Est-ce l'utilisateur qui invoque explicitement (/test, /deploy), ou l'agent qui juge lui-même que c'est pertinent en lisant la description ?

User-triggered

Slash command, invocation explicite

L'utilisateur tape /cmd et l'agent exécute. Pas d'ambiguïté, ça part toujours à la demande. Bon pour les workflows ponctuels qu'on ne veut pas voir se déclencher tout seul.

Workflows, Commands, Prompt files

⇆
Agent-triggered

L'agent juge lui-même la pertinence

L'agent lit les descriptions disponibles et décide tout seul, à chaque tour, si l'une d'elles correspond à la tâche. Pas besoin de penser à l'invoquer, mais ça suppose des descriptions bien rédigées.

Skills, Agent Requested rules (Cursor)

Exemple parlant

Pourquoi « l'agent décide » est plus économe

Imaginons une règle pour nommer mes fichiers en camelCase. Si elle est toujours active, elle entre dans le contexte à chaque tour, y compris quand je demande à l'agent d'éditer une classe (où la règle ne sert à rien).

Si c'est l'agent qui décide, il lit la description (« règle de nommage de fichiers ») et la charge uniquement quand on parle de fichiers. Pour ma classe, il l'ignore. Multiplié par toutes les règles du projet, l'économie de tokens est massive.

C'est exactement l'idée des Rules à activation conditionnelle : chez Antigravity, une Rule se règle en Always On / Manual / Model Decision ; chez Cursor, le mode « Agent Requested » fait pareil — c'est le LLM qui juge, sur description, si la règle s'applique.

03

Fichier monolithique / Progressive disclosure

Quand le contenu est chargé, est-ce qu'il l'est en un bloc, ou par paliers, en chargeant seulement les sous-parties pertinentes ? L'axe 1 disait si le contenu entre d'office ; celui-ci dit comment il est découpé une fois qu'on le charge.

Monolithique

Tout ou rien

Le fichier est chargé en entier. Simple à comprendre, mais quand le fichier devient gros, on consomme des tokens même pour les parties non utilisées dans la tâche en cours.

CLAUDE.md, .cursorrules legacy

⇆
Progressive

En 3 niveaux

D'abord les métadonnées, puis le SKILL.md, puis les fichiers référencés à la demande. Le contenu est découpé en paliers plutôt que chargé en bloc : on lit la sous-partie utile, pas le fichier entier.

Skills (toutes implémentations)

Avec ces trois axes, vous pouvez maintenant cataloguer n'importe quel mécanisme proposé par n'importe quel harnais. Place au panorama.

06 Outil par outil

Le panorama complet

Chaque harnais a sa propre palette de mécanismes. Pour les comparer, on les classe selon les couleurs des quatre familles : Rules (toujours actifs), Skills (à la demande, agent décide), Workflows (à la demande, user décide), Hooks (déterministes). Dans chaque carte, ce sont les pastilles qui portent la couleur de famille ; la fine bordure du haut n'est qu'un accent par éditeur. Les Hooks, d'une nature à part, ont droit à leur propre chapitre juste après.

07 La famille à part

Les Hooks — quand rien ne doit être laissé au modèle

On les a aperçus dans chaque carte du panorama ; il est temps de leur rendre justice. Tous les autres mécanismes partagent un point commun : ils influencent le LLM, qui reste libre de décider. Les Hooks, eux, sont d'une autre nature. Ce n'est pas du contexte, c'est du code déterministe qui s'exécute autour des actions de l'agent — qu'il le veuille ou non. Voilà pourquoi ils méritent leur propre chapitre.

Skills · Rules · Commands

Influencer le modèle

On suggère au LLM quoi faire, via du contexte. Bien rédigé, il suit. Mais il garde son libre arbitre : il peut interpréter, contourner, oublier. C'est probabiliste.

Hooks

Contraindre la mécanique

On force un comportement par du code, en dehors du LLM. Le hook s'exécute toujours, à un moment précis du cycle, et peut bloquer ou modifier l'action. C'est déterministe.

Même cycle de vie, trois jeux de noms. Copilot et Claude Code partagent presque le même vocabulaire (camelCase vs PascalCase) ; OpenCode, lui, expose un « bus d'événements ».

Exemple concret — interdire toute écriture dans /legacy
1 · on déclare le hook — .claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": ".claude/guard-legacy.sh"
      }]
    }]
  }
}
2 · le script — .claude/guard-legacy.sh
#!/usr/bin/env bash
# reçoit l'appel d'outil en JSON sur stdin
input=$(cat)
path=$(echo "$input" | jq -r '.tool_input.file_path')

if [[ "$path" == *"/legacy/"* ]]; then
  echo "/legacy est protégé." >&2
  exit 2   # ≠ 0 → le harnais bloque
fi
exit 0     # 0 → l'action se poursuit
Le hook est déclenché par la configuration, pas par le LLM. Avant chaque Write ou Edit, le harnais exécute le script en lui passant le détail de l'action. Le script décide : exit 0 laisse passer, exit 2 bloque.
Ce qui se passe quand l'agent tente l'interdit
Agent « Je corrige le bug dans legacy/auth.js » → tente Write(legacy/auth.js)
Hook PreToolUse déclenche guard-legacy.sh → le chemin contient /legacy/ → exit 2
Harnais ⛔ appel Write bloqué — le fichier n'est pas touché. L'agent ne peut pas passer outre.
Agent reçoit le retour d'erreur (/legacy est protégé), s'adapte : propose de corriger le bug ailleurs, ou prévient l'utilisateur.
Le cas d'usage roi des hooks : les garde-fous. Tout ce qui ne doit jamais dépendre du bon vouloir du modèle — secrets jamais lus, fichiers protégés, formatage imposé, journalisation obligatoire — se câble en hook et devient infranchissable. Là où une Rule ou un Skill suggèrent, le hook garantit.

Retenez la ligne de partage : tout le reste guide le LLM, les Hooks court-circuitent son jugement. C'est la seule famille qui ne parie pas sur la coopération du modèle.

08 Pour conclure

La règle qui explique tout

Vous avez maintenant traversé un zoo de noms : Skills, Rules, Workflows, Commands, Instructions, Custom Agents, Hooks, Guidelines, Conventions… et derrière, à chaque fois, le même besoin et la même solution.

Tous ces mécanismes sont des manières de pré-charger du contexte dans le harnais, sans avoir à le retaper à chaque conversation. La compétition entre éditeurs se joue sur l'ergonomie (la lourdeur du fichier, la granularité du déclenchement, l'économie de tokens, la composabilité), pas sur le concept.

L'innovation pédagogiquement intéressante de ces derniers mois, c'est la progressive disclosure popularisée par les Skills d'Anthropic et reprise par Antigravity, Codex et Gemini CLI : permettre à l'agent de juger lui-même ce dont il a besoin, plutôt que d'imposer le contexte d'office. C'est cette idée qui change le plus l'expérience à grande échelle.
